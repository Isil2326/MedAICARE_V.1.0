**PHASE 4 — MOTEUR DE RECOMMANDATION OPEN-LOOP**



Objectif général :

Construire un moteur de recommandations non automatisé, non prescriptif, explicable, traçable et validé par clinicien, à partir des prédictions et du contexte temporel, sans jamais produire de dose, décision thérapeutique automatique ou action clinique autonome.



La Phase 4 ne doit pas créer un système fermé. Elle doit créer un système de suggestions open-loop, soumises à validation humaine.



**Contraintes non négociables Phase 4 :**



**1. Open-loop strict**

\- Toute recommandation doit rester au statut `pending`.

\- Aucune recommandation ne doit être automatiquement appliquée.

\- Aucune modification de traitement.

\- Aucune dose d’insuline.

\- Aucune consigne thérapeutique prescriptive.

\- Validation ou rejet clinicien obligatoire.



**2. Données synthétiques uniquement**

\- Toutes les recommandations générées doivent rester `is\_synthetic=True`.

\- Aucun usage de données réelles.

\- Aucune revendication clinique réelle.



**3. XAI = support uniquement**

\- Le moteur peut joindre des explications XAI pour affichage/audit.

\- La XAI ne doit jamais être la condition principale de décision.

\- Si `xai\_reliability\_status = not\_reliable\_for\_clinical\_interpretation`, alors :

&#x20; - ne pas utiliser l’explication comme justification clinique ;

&#x20; - ajouter un warning ;

&#x20; - éventuellement générer une recommandation plus prudente ou uniquement “surveillance/clinician\_review”.

\- Le moteur doit lire et respecter `xai\_reliability\_status`.



**4. Recommandations non prescriptives**

Interdire :

\- “injectez X unités” ;

\- “augmentez votre dose” ;

\- “diminuez votre traitement” ;

\- “modifiez votre basal” ;

\- “prenez tel traitement” ;

\- toute décision thérapeutique automatisée.



Autoriser uniquement :

\- suggestions de vigilance ;

\- recommandations de surveillance ;

\- demande de confirmation clinique ;

\- alerte à discuter avec clinicien ;

\- message d’éducation non prescriptif ;

\- orientation vers consultation / contact professionnel si scénario critique synthétique.



**5. Validation clinicien obligatoire**

Workflow minimal :

\- `pending`;

\- `approved`;

\- `rejected`;

\- optionnel : `modified`.



Chaque action doit être auditée.



**6. Traçabilité complète**

Chaque recommandation doit tracer :

\- patient\_id ;

\- prediction\_id ;

\- model\_id ;

\- probability ;

\- target ;

\- horizon ;

\- trigger ;

\- features principales ;

\- xai\_reliability\_status ;

\- rule\_id ;

\- rule\_version ;

\- rationale ;

\- safety\_level ;

\- status ;

\- reviewed\_by ;

\- reviewed\_at ;

\- review\_note ;

\- is\_synthetic.



**7. Pas de mobile**

\- Ne pas commencer React Native Expo.

\- Ne pas créer d’interface mobile.

\- Produire uniquement backend/API/documentation/tests.



**8. Pas de nouvelle modélisation ML**

\- Ne pas réentraîner les modèles sauf nécessité technique documentée.

\- Ne pas modifier les performances ML.

\- Ne pas modifier le benchmark.

\## 2. Types de recommandations



Implémenter les catégories suivantes, en restant non prescriptif :



\- `ALERT\_CRITICAL`

\- `RECOMMENDATION\_BEHAVIORAL`

\- `CLINICAL\_REFERRAL`

\- `THERAPY\_SUGGESTION\_REVIEW\_ONLY`



Définition importante :



`THERAPY\_SUGGESTION\_REVIEW\_ONLY` ne doit jamais contenir une dose ou une instruction de changement de traitement. Elle signifie uniquement :

“élément à revoir avec un clinicien”.



\## 3. DSL / règles expertes



Créer un système de règles simple, versionné et traçable.



Chaque règle doit contenir :



\- `rule\_id`;

\- `rule\_version`;

\- `name`;

\- `description`;

\- `trigger\_condition`;

\- `target`;

\- `horizon\_min`;

\- `probability\_threshold`;

\- `required\_context`;

\- `safety\_constraints`;

\- `message\_template`;

\- `category`;

\- `priority`;

\- `evidence\_level`;

\- `open\_loop\_notice`.



Exemple de règles non prescriptives :



\### Hypo risk 30/60



Si :

\- target = hypo ;

\- probability >= seuil ;

\- horizon = 30 ou 60 ;



Alors :

\- catégorie : `ALERT\_CRITICAL` ou `RECOMMENDATION\_BEHAVIORAL` selon seuil ;

\- message patient prudent :

&#x20; “Risque indicatif d’hypoglycémie détecté. Vérifiez votre glycémie selon vos pratiques habituelles et contactez un professionnel en cas de symptômes.”

\- aucune dose ;

\- aucune instruction thérapeutique.



\### Hyper risk 30/60



Si :

\- target = hyper ;

\- probability >= seuil ;



Alors :

\- message non prescriptif :

&#x20; “Risque indicatif d’hyperglycémie. Surveillez l’évolution et discutez avec votre soignant si cela persiste.”

\- pas d’ajustement insulinique automatique.



\### XAI low reliability



Si :

\- `xai\_reliability\_status = not\_reliable\_for\_clinical\_interpretation`;



Alors :

\- ne pas générer de justification XAI clinique ;

\- ajouter warning ;

\- éventuellement catégorie : `CLINICAL\_REFERRAL` ou `REVIEW\_ONLY`;

\- message :

&#x20; “L’explication du modèle est jugée peu fiable pour interprétation clinique ; validation professionnelle requise.”



\## 4. Safety layer



Créer une couche de sécurité obligatoire avant persistance.



Elle doit vérifier :



\- absence de dose ;

\- absence de termes interdits ;

\- absence d’ordre thérapeutique ;

\- présence de disclaimer open-loop ;

\- présence de `is\_synthetic=True`;

\- présence d’un statut `pending`;

\- présence d’un `rule\_id`;

\- présence d’un `rationale`;

\- conformité de la catégorie ;

\- xai reliability respectée.



Termes interdits :

\- injectez ;

\- prenez X unités ;

\- augmentez votre dose ;

\- diminuez votre dose ;

\- modifiez votre traitement ;

\- arrêtez ;

\- commencez ;

\- prescription ;

\- traitement recommandé ;

\- vous devez.



Si un message contient un terme interdit :

\- rejet de génération ;

\- audit `recommendation.safety\_blocked`.

\## 6. Workflow open-loop



Étendre ou utiliser la table `recommendations`.



États minimum :

\- `pending`;

\- `approved`;

\- `rejected`.



Option souhaitable :

\- `modified`.



Transitions autorisées :



\- generated → pending ;

\- pending → approved ;

\- pending → rejected ;

\- pending → modified ;

\- modified → approved ;

\- modified → rejected.



Interdits :

\- approved automatique ;

\- rejected automatique sans acteur ;

\- action thérapeutique automatique.



Chaque transition doit être auditée.

Patient :



&#x20;   peut lire ses recommandations approuvées uniquement ;

&#x20;   ne peut pas générer ;

&#x20;   ne peut pas approuver/rejeter ;

&#x20;   ne peut pas voir les recommandations pending si jugé sensible.



Clinicien :



&#x20;   peut générer ;

&#x20;   peut voir pending ;

&#x20;   peut approuver/rejeter/modifier ;

&#x20;   peut voir les dossiers selon RBAC prototype.



Admin :



&#x20;   idem clinicien + audit.



Non authentifié :



&#x20;   aucun accès.



Si pour prototype le patient lit aussi pending, le documenter explicitement. Mais préférence : patient voit uniquement approved.
