Phase 0.1 — Hardening minimal avant pipeline temporel

À compléter avant ou au tout début de la Phase 1 :

1. Rate limiting minimal

Ajouter un rate limiting sur :
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

Exigence :
- solution simple acceptable pour prototype Replit ;
- en mémoire si instance unique ;
- Redis-ready si possible ;
- réponses 429 documentées ;
- tests associés :
  - trop de tentatives login → 429 ;
  - refresh abusif → 429 ;
  - comportement normal non bloqué.

2. Endpoint `/ready`

Ajouter :
- `GET /ready`

Il doit vérifier au minimum :
- connexion DB ;
- état migration compatible ou table critique accessible ;
- retour clair :
  - `200 ready`
  - ou `503 not ready`.

Ajouter test associé.

3. Smoke test PostgreSQL / Alembic

Même si les tests unitaires restent sur SQLite jetable, ajouter au minimum une commande ou un script documenté permettant de vérifier sur PostgreSQL réel :
- `alembic upgrade head`;
- présence des 14 tables ;
- présence des contraintes critiques ;
- présence des index `(patient_id, ts)` ;
- présence des contraintes uniques audit `sequence` et `entry_hash`.

Objectif : éviter que le socle soit validé uniquement par SQLite.

4. OpenAPI examples

Ajouter des exemples explicites `json_schema_extra` ou équivalent pour :
- register patient ;
- register clinician ;
- login ;
- refresh ;
- approve recommendation ;
- reject recommendation.

Ce n’est pas critique fonctionnellement, mais utile pour documentation, soutenance et tests manuels.

5. Politique mot de passe

La règle actuelle min 8 est acceptable pour prototype, mais je veux au minimum documenter clairement :
- politique actuelle ;
- limite ;
- amélioration prévue :
  - complexité ;
  - dictionnaire ;
  - contrôle fuite ;
  - verrouillage temporaire après échecs.

Si possible, ajouter déjà une règle minimale :
- min 10 ou 12 caractères ;
- au moins une lettre ;
- au moins un chiffre ;
- au moins un caractère spécial.

6. Audit concurrency note

Documenter dans le README ou dans `docs/security` la limite potentielle du chaînage audit en cas d’insertions concurrentes.

Si possible, renforcer l’append audit par transaction stricte ou verrou applicatif léger afin d’éviter une course sur `sequence`.

Après ces corrections, produire un court amendement :

- fichiers modifiés ;
- tests ajoutés ;
- sortie `pytest`;
- confirmation que les endpoints `/health` et `/ready` fonctionnent ;
- confirmation que le rate limiting fonctionne ;
- confirmation que les migrations PostgreSQL sont rejouables.

Une fois cette Phase 0.1 terminée, je valide le passage à :

Phase 1 — Data Engineering & Pipeline temporel.

Objectif de Phase 1 :
construire un pipeline temporel médical robuste, anti-leakage, prêt pour ML/XAI, en s’appuyant sur le socle backend validé.

Périmètre Phase 1 attendu :

1. TimescaleDB / PostgreSQL temporel

- Tenter l’activation TimescaleDB si disponible.
- Si TimescaleDB n’est pas disponible sur Replit, documenter clairement le fallback PostgreSQL standard.
- Préparer les migrations de manière compatible :
  - hypertables si extension disponible ;
  - index temporels renforcés sinon ;
  - aucune dépendance silencieuse non documentée.

Tables concernées :
- `cgm_readings`
- `insulin_events`
- `meal_events`
- `activity_events`
- `predictions`
- `recommendations`

2. Timestamps médicaux

Imposer :
- stockage UTC ;
- champs `ts` timezone-aware ;
- validation Pydantic ;
- interdiction des timestamps naïfs ;
- gestion des dates futures aberrantes ;
- gestion ordre temporel.

3. Endpoints ingestion temporelle

Créer des endpoints API sécurisés pour données simulées :

- `POST /api/v1/timeseries/cgm`
- `POST /api/v1/timeseries/insulin`
- `POST /api/v1/timeseries/meals`
- `POST /api/v1/timeseries/activity`
- `GET /api/v1/timeseries/cgm`
- `GET /api/v1/timeseries/events`

Contraintes :
- patient : peut écrire/lire uniquement ses propres données simulées ;
- clinicien/admin : lecture autorisée selon droits ;
- toute écriture doit être auditée ;
- toutes les données restent `is_synthetic=True` par défaut à ce stade.

4. Validation Pydantic stricte

Ajouter des schémas d’entrée avec :
- bornes physiologiques ;
- unités explicites ;
- champs requis ;
- validation timestamp ;
- validation `patient_id`;
- rejet valeurs impossibles.

Exemples de bornes minimales :
- CGM : 30–600 mg/dL ;
- insuline : > 0 et limite max documentée ;
- glucides : >= 0 et limite max documentée ;
- activité : durée positive ;
- horizon prédiction : 30 ou 60 minutes uniquement dans cette phase.

5. Déduplication et idempotence

Mettre en place une stratégie de déduplication :
- clé logique possible : `patient_id + ts + source/device_id + value`;
- ou `external_event_id`;
- éviter doubles insertions ;
- documenter les choix.

Ajouter tests :
- double POST même événement ;
- comportement attendu ;
- pas de duplication silencieuse.

6. Données manquantes et anomalies

Préparer la logique :
- détection valeurs hors plage ;
- détection gaps CGM ;
- détection timestamps incohérents ;
- marquage qualité de données si nécessaire.

Si le modèle actuel ne contient pas encore `quality_flag`, `source`, `external_event_id`, proposer une migration additive.

7. Préparation feature engineering

- rolling mean ;
- rolling std ;

- dG/dt ;
- indicateurs post-prandiaux ;
- indicateurs post-insuline ;
- TIR glissant ;

- hour of day ;
- day of week ;
- flags nocturnes.

Livrable attendu :
- fonctions pures testables ;
- pas encore besoin de modèle ML ;
- tests unitaires sur séries simulées.

8. Anti temporal leakage

- aucune feature à l’instant T ne doit utiliser une donnée future ;
- fenêtres glissantes strictement passées ;
- tests explicites anti-leakage ;
