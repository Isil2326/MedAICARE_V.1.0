\- Phase 2 — Modélisation IA/ML ;

Le blocage scientifique identifié en Phase 2 est levé dans le cadre strictement synthétique :



\- les 4 couples sont désormais évaluables sur le test ;

\- chaque couple possède positifs et négatifs dans train / validation / test ;

\- le test contient ≥ 10 positifs pour chaque couple ;

\- les métriques sont calculées réellement ;

\- aucune dose n’est suggérée ;

\- aucune notification clinique automatique n’est déclenchée ;

\- les données restent 100 % synthétiques `is\_synthetic=True`.



Les performances très élevées sont acceptées uniquement comme preuve de cohérence du pipeline sur un benchmark synthétique scénarisé. Elles ne doivent jamais être présentées comme performances cliniques transférables.

**PHASE 3 — XAI CLINIQUE**



Objectif général :

Construire une couche d’explicabilité exploitable, testable et documentée pour les prédictions ML, en distinguant clairement :

\- explications techniques pour cliniciens ;

\- explications simplifiées pour patients ;

\- importance statistique ;

\- absence de causalité ;

\- limites liées aux données synthétiques.



La Phase 3 doit expliquer les prédictions, pas produire des décisions thérapeutiques.



Contraintes non négociables Phase 3 :



**1. Données synthétiques uniquement**

\- Toutes les explications sont générées sur données synthétiques.

\- Aucun usage de données patient réelles.

\- Aucun claim clinique réel.



**2. Open-loop strict**

\- Les explications accompagnent une probabilité.

\- Elles ne recommandent pas une dose.

\- Elles ne modifient pas un traitement.

\- Elles ne déclenchent aucune action automatique.



**3. XAI ≠ causalité**

\- Documenter explicitement que SHAP/LIME/EBM expliquent le comportement du modèle, pas les causes médicales réelles.

\- Aucun message ne doit dire “la cause est”.

\- Utiliser plutôt :

&#x20; - “le modèle a principalement pondéré…”

&#x20; - “ce facteur contribue au score selon le modèle…”

&#x20; - “interprétation indicative sur données simulées”.



**4. Modèles calibrés**

\- Les modèles actifs sont calibrés.

\- Il faut documenter précisément ce qui est expliqué :

&#x20; - sortie brute du modèle ;

&#x20; - probabilité calibrée ;

&#x20; - ou pipeline complet modèle + calibration.

\- Si SHAP/LIME expliquent le modèle non calibré, le préciser clairement.

\- Si possible, expliquer la probabilité finale calibrée via wrapper, mais ne pas masquer les limites.



**5. Pas de Phase 4**

\- Ne pas commencer le moteur de recommandation thérapeutique.

\- Ne pas créer de règles de décision clinique.

\- Ne pas générer de recommandations comportementales.

\- La Phase 3 est uniquement XAI.



**6. Pas de mobile**

\- Ne pas commencer React Native Expo.

\- Produire des API et structures prêtes pour futur affichage mobile/web, mais pas d’interface mobile complète.

\## 2. Méthodes XAI obligatoires



Implémenter selon compatibilité modèle :



\### EBM



Pour les modèles EBM :

\- utiliser les explications natives InterpretML ;

\- global feature importance ;

\- local contribution ;

\- courbes ou données de contribution par feature si accessibles.



\### XGBoost / RandomForest



Pour les modèles tree-based :

\- utiliser SHAP TreeExplainer si disponible ;

\- sinon fallback documenté.



\### Logistic Regression



Pour LogReg :

\- utiliser coefficients normalisés ;

\- éventuellement SHAP LinearExplainer si disponible.



\### LIME



Pour tous les modèles compatibles :

\- LIME tabular local ;

\- configuration documentée ;

\- seed fixe ;

\- nombre de perturbations contrôlé ;

\- limites de stabilité documentées.



Important :

ne pas forcer une méthode qui ne fonctionne pas proprement sur Replit. Si une méthode échoue, fournir fallback explicite + documentation honnête.

\## 8. Évaluation XAI



Implémenter des métriques d’évaluation XAI, au moins sur un sous-échantillon synthétique.



Obligatoire :

Fidélité locale



&#x20;   Pour LIME : score de fidélité locale / R² si disponible.

&#x20;   Pour SHAP : vérifier cohérence additive si possible.



Stability



&#x20;   Perturber légèrement les features ;

&#x20;   recalculer explications ;

&#x20;   mesurer stabilité du top-k ;

&#x20;   Spearman ou overlap top-k.



Deletion / perturbation test



&#x20;   retirer ou neutraliser les top features ;

&#x20;   vérifier que la sortie modèle varie davantage qu’en retirant des features faibles.



Agreement SHAP / LIME / native



&#x20;   top-k overlap ;

&#x20;   corrélation de rang si possible ;

&#x20;   divergences documentées.



Congruence physiologique



Tests heuristiques :



&#x20;   une glycémie moyenne élevée devrait contribuer au risque hyper plutôt qu’hypo ;

&#x20;   une glycémie basse / pente descendante devrait contribuer au risque hypo ;

&#x20;   les interprétations absurdes doivent être détectées et documentées.



Ne pas prétendre que cette congruence vaut validation clinique réelle.
