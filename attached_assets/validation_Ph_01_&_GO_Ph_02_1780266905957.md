PHASE 2 — MODÉLISATION IA/ML

Objectif général :
Construire une première couche ML scientifique, reproductible et défendable pour la prédiction hypo/hyperglycémie à horizon 30–60 minutes, sans prétendre à une validation clinique réelle et sans automatiser de décision thérapeutique.

Contraintes non négociables Phase 2 :

1. Données simulées uniquement
- Ne pas introduire de données réelles.
- Ne pas prétendre à une performance clinique réelle.
- Toutes les données issues du seed ou générées doivent rester identifiées comme synthétiques.
- Si un dataset externe réel n’est pas disponible légalement et proprement, ne pas l’inventer.

2. Aucune métrique inventée
- Toute métrique affichée doit provenir d’un calcul réel sur un jeu de test identifié.
- Interdiction de coder en dur AUC, F1, recall, précision ou toute autre métrique.
- Si la taille du dataset est insuffisante, le signaler explicitement.

3. Open-loop strict
- Le modèle produit des probabilités ou scores.
- Il ne produit pas de décision thérapeutique automatique.
- Il ne modifie aucun traitement.
- Les recommandations resteront une phase ultérieure ou un placeholder clairement identifié.

4. Pas de mobile en Phase 2
- Ne pas commencer React Native Expo.
- Ne pas modifier le frontend mobile.
- La phase se concentre sur ML backend, pipelines, scripts, tests, documentation.

5. Pas de XAI clinique avancée en Phase 2
- SHAP/LIME complets sont prévus en Phase 3.
- En Phase 2, vous pouvez préparer les interfaces et stocker les artefacts nécessaires, mais ne pas présenter d’explications SHAP/LIME comme validées si elles ne le sont pas.

6. Anti-leakage obligatoire
- Toute construction de labels et features doit respecter strictement le temps.
- À un instant T, les features ne doivent utiliser que les données `ts <= T`.
- Les labels peuvent regarder dans le futur uniquement pour construire la vérité terrain du dataset d’entraînement, jamais pour les features.
- Des tests anti-leakage doivent être ajoutés ou étendus.

## 2. Définition claire du problème prédictif

Définir explicitement les tâches :

### Tâche 1 — Hypoglycémie future

Prédire le risque qu’un patient atteigne une glycémie `< 70 mg/dL` dans les prochaines :

- 30 minutes ;
- 60 minutes.

Optionnel :
- hypo sévère `< 54 mg/dL`, uniquement si assez de données simulées.

### Tâche 2 — Hyperglycémie future

Prédire le risque qu’un patient atteigne une glycémie `> 180 mg/dL` dans les prochaines :

- 30 minutes ;
- 60 minutes.

Optionnel :
- hyper sévère `> 250 mg/dL`, uniquement si assez de données.

Les seuils doivent être documentés comme seuils de simulation/inspiration clinique, non comme validation médicale.

## 3. Construction des labels

Créer une logique de labellisation temporelle :

Pour chaque instant T :
- features calculées avec données `<= T`;
- label hypo_30 = 1 si une lecture CGM `< 70` apparaît dans `(T, T+30min]`;
- label hypo_60 = 1 si une lecture CGM `< 70` apparaît dans `(T, T+60min]`;
- label hyper_30 = 1 si une lecture CGM `> 180` apparaît dans `(T, T+30min]`;
- label hyper_60 = 1 si une lecture CGM `> 180` apparaît dans `(T, T+60min]`.

Ajouter tests :
- label positif attendu ;
- label négatif attendu ;
- label ignore les données avant T ;
- features ne changent pas si on ajoute des données futures ;
- absence de fuite temporelle.

## 5. Splits temporels

Implémenter des splits strictement temporels :

- train ;
- validation ;
- test.

Interdit :
- split aléatoire naïf mélangeant le futur et le passé ;
- fuite patient/temps non documentée.

Stratégie recommandée :
- split par temps à l’intérieur de chaque patient ;
- ou split par période globale ;
- documenter le choix.

Ajouter une fonction de validation :
- aucune date du train ne doit dépasser la date minimale du test si split global ;
- si split par patient, documenter précisément les garanties.
